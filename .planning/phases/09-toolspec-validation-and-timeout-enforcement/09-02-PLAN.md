# Wave 2: Execution Constraints & UI Feedback

## Objective
Enforce execution payload strictness by rejecting unknown flags (HTTP 400), implement graceful timeout termination via the Podman API (`SIGTERM` -> 5s -> `SIGKILL`), and wire up inline form validation feedback in the React frontend.

## Tasks

```xml
<task>
  <action>Update `internal/api/execute.go` to explicitly validate incoming job execution keys. Loop over the submitted JSON payload (`req.Values`), and if any key is NOT defined in `ToolSpec.Flags[].ID` or `ToolSpec.Inputs[].ID`, return an HTTP 400 Bad Request containing a JSON payload mapping the invalid field to "Unknown flag".</action>
  <read_first>
    - internal/api/execute.go
    - internal/models/toolspec.go
  </read_first>
  <acceptance_criteria>
    - `execute.go` loops over request values and verifies their existence in `spec.Flags` or `spec.Inputs`
    - Submitting an unknown flag returns HTTP 400
  </acceptance_criteria>
</task>

<task>
  <action>Update `internal/runtime/podman.go` to implement graceful container timeouts. Instead of a hard `context.WithTimeout` killing the HTTP request, start a goroutine that waits on `time.After(time.Duration(spec.Metadata.Runtime.TimeoutSeconds) * time.Second)`. When the timer fires, use the Podman bindings `containers.Kill(ctx, containerID, &containers.KillOptions{Signal: pointer("SIGTERM")})`. Then wait 5 seconds. If the container is still running, call `containers.Kill` again with `Signal: pointer("SIGKILL")`.</action>
  <read_first>
    - internal/runtime/podman.go
  </read_first>
  <acceptance_criteria>
    - `podman.go` implements a timeout timer based on `spec.Metadata.Runtime.TimeoutSeconds`
    - Sends `SIGTERM` when timeout expires
    - Sends `SIGKILL` 5 seconds after `SIGTERM` if container has not exited
  </acceptance_criteria>
</task>

<task>
  <action>Update the React frontend (e.g. `web/src/components/ExecutionForm.tsx` or similar execution/draft UI) to catch HTTP 400 JSON error responses from the backend. Maintain an `errors` object in state mapping field IDs to error messages, and dynamically render these messages inline (e.g. as red text below the offending input field) rather than relying exclusively on a global banner.</action>
  <read_first>
    - web/src/pages/ExecutionPage.tsx
  </read_first>
  <acceptance_criteria>
    - The execution form captures HTTP 400 validation error payloads
    - Field-specific errors are displayed inline next to the respective inputs
  </acceptance_criteria>
</task>
```
